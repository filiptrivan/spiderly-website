## Translating Entities and Properties

Spiderly uses JSON translation files to customize how entity and property names appear across the app:
- **Customizing labels** (e.g., by default a text input shows `ProductName` as the label — adding a translation changes it to `Product Name`)
- **Excel export filenames**
- **Validation messages**

### How It Works

Translation files live in `Backend\YourAppName.Shared\Translations\`. Each file is named by [language tag](https://en.wikipedia.org/wiki/IETF_language_tag#List_of_common_primary_language_subtags) (e.g., `en.json`, `sr-Latn-RS.json`, `de.json`).

On build, the `TranslationsGenerator` source generator:
1. Scans your entities to discover all expected translation keys
2. Auto-scaffolds missing keys with empty values into each JSON file
3. Generates `TermsGenerated.generated.cs` (backend translations)
4. Generates `{lang}.generated.json` files for Angular i18n

### Translation Keys

For each entity, three keys are generated:
- `{EntityName}` — singular name (used in forms, labels)
- `{EntityName}List` — plural name (used in table titles)
- `{EntityName}ExcelExportName` — Excel export filename (falls back to plural name)

For each property:
- `{PropertyName}` — used as input field labels and in validation messages

### Example

Given this entity:

```csharp
public class UserExtended : BusinessObject<long>
{
    public string Email { get; set; }
}
```

Create `Backend\YourAppName.Shared\Translations\en.json`:

```json
{
  "Email": "Email address",
  "UserExtended": "User",
  "UserExtendedExcelExportName": "Users",
  "UserExtendedList": "Users"
}
```

And `Backend\YourAppName.Shared\Translations\sr-Latn-RS.json`:

```json
{
  "Email": "Email adresa",
  "UserExtended": "Korisnik",
  "UserExtendedExcelExportName": "Korisnici",
  "UserExtendedList": "Korisnici"
}
```

<Callout>
You don't need to create these keys manually. Just create the JSON file (even empty `{}`), build your project, and the generator will auto-scaffold all expected keys with empty values. Then fill in the translations.
</Callout>

### Adding a New Language

1. Create a new file, e.g., `Backend\YourAppName.Shared\Translations\de.json` (can be empty `{}`)
2. Build the project — the generator auto-scaffolds all keys with empty values
3. Fill in the German translations
4. Build again — translations are available on both backend and frontend

### Excel Export Filename

The `{EntityName}ExcelExportName` key controls the Excel export filename. If left empty, it falls back to `{EntityName}List`. If that's also empty, the entity name is used as-is.

## Changing Default Language

By default, Spiderly uses English (`en`) as the primary language. If you want to change the default language, you'll need to update the language tag in both your **backend** and **frontend** configurations.

### Language Tag Update:

1. Open the `Backend\YourAppName.WebAPI\Startup.cs` file and update the `SpiderlyConfigureServices` call by passing your desired language tag:

```csharp
public class Startup
{
    public void ConfigureServices(IServiceCollection services)
    {
        services.SpiderlyConfigureServices<YourAppNameApplicationDbContext>("your-language-tag");
    }
}
```

2. Open the `Frontend\src\app\app.config.ts` file and update the `provideSpiderlyTransloco` configuration:

```typescript
export const appConfig: ApplicationConfig = {
  providers: [
    provideSpiderlyTransloco({
      preloadLangs: ['your-language-tag'],
      availableLangs: [
        'your-language-tag',
        'your-language-tag.generated',
      ],
      defaultLang: 'your-language-tag',
      fallbackLang: 'your-language-tag.generated',
    }),
  ]
};
```

## Custom Frontend Translations for Multilingual Apps

<Callout>
You only need to add custom frontend translations if your app supports **multiple languages**. If you're only using a single language, you can simply use hardcoded strings.
</Callout>

### Add a Translation Key-Value Pair

Open the appropriate language file under `Frontend\src\assets\i18n` folder, for example: `en.json`.

Then, add a new key-value entry like this anywhere in the file: `"KeyForTranslation": "Translated text"`.

You can now reference this key in your TypeScript or HTML files.

### Use Translation in TypeScript (`.ts`)

Inject the `TranslocoService` and use its `translate` method:

```typescript
// ... other imports
import { TranslocoService } from '@jsverse/transloco';

@Component({
  templateUrl: './your.component.html',
})
export class YourComponent {

  constructor(
    // ... other injections
    private translocoService: TranslocoService
  ) {}

  ngOnInit() {
    const translatedText = this.translocoService.translate('KeyForTranslation');
  }
}
```

### Use Translation in HTML (`.html`)

Add `TranslocoDirective` into Angular component imports:

```typescript
@Component({
  templateUrl: './your.component.html',
  imports: [
    // ... other imports
    TranslocoDirective,
  ],
})
export class YourComponent {
  // ...
}
```

Use the directive with Angular's `*transloco` syntax:

```html
<ng-container *transloco="let t">
    {{t('KeyForTranslation')}}
</ng-container>
```

## Custom Backend Translations for Multilingual Apps

<Callout>
You only need to add custom backend translations if your app supports **multiple languages**. If you're only using a single language, you can simply use hardcoded strings.
</Callout>

### Use Case Example

Let's say you want to throw a translated exception message, based on the active culture (e.g., English):

```csharp
throw new Exception(Terms.YourExceptionKey);
```

If that's the case, open the `Backend\YourAppName.Shared\Resources\Terms.resx` file and add a new entry with the key `YourExceptionKey` and a culture-specific value, for example: `Your exception message.`

Spiderly will automatically pick the correct translation at runtime based on the active culture settings.

<Callout>
For easier management of translation keys across multiple `.resx` files, you can use [ResXManager](https://marketplace.visualstudio.com/items?itemName=TomEnglert.ResXManager) - a Visual Studio extension that simplifies working with localization resources.
</Callout>
